# Scholar-RAG 落地实践研究与工程演进报告
**副标题：从理论拓扑到生产级基础设施的渐进式演变方案**

本报告基于《Scholar-RAG 学术系统架构全景宪法论文》，以“理想架构与工程现实的折中（Trade-off）”为核心哲学，将高阶的非线性检索、多模态反思与知识流形理论，降维拆解为5个阶段的可执行落地计划。

**核心原则：绝不搞“大爆炸式发布（Big Bang Release）”，所有技术栈引入必须具有“成本可控、可降级、可热插拔”的现实特征。**

---

## 阶段 0：基础设施定点与微服务契约定义 (Infrastructure & API Contracts)
**理论对应**：学术宪法中的“生命周期基质”与“防浪堤”。
**现实约束**：初期团队与算力有限，无法一次性拉起几十个微服务，数据库维护成本高。

*   **方向与动作**：
    *   **基建选型闭环**：确立数据库“三驾马车”：PostgreSQL（主业务流形/用户态数据）、Milvus（高维 Dense/Sparse 向量双路基质）、Neo4j（拓扑知识图谱关系基质）。
    *   **接口契约 (API Contracts) 先行**：由后端定义标准化的 JSON 输入输出结构（如宪法第17章的态矢量规范），前后端分离开发，解耦 UI 渲染与大模型流式推理（SSE）。
*   **现实妥协（极简防御）**：
    *   初期不强求物理级别的差分隐私或硬件级数据库隔离，而是实行**逻辑隔离**（为所有向量和图节点强绑定 `tenant_id` 和 `doc_id` 标签）。
    *   本地开发优先使用 Docker Compose 启动单机全栈环境，轻量跑通。

## 阶段 1：高保真知识摄入管线 (High-Fidelity Knowledge Ingestion)
**理论对应**：相态 1：全息 PDF 剥离与多模态降维同构。
**现实约束**：大语言模型的原生 OCR（如 Qwen-VL）极度消耗 Token 和算力，每页都用大模型解析会导致摄入成本爆炸，时间难以忍受。

*   **方向与动作**：
    *   **组装分级 Parser Matrix**：将解析器分层。纯文本页使用规则基解析（如 PyPDF/pdfplumber 变种）；含有复杂表格、多栏、数学公式的页面，触发降级重试机制，转入 Marker 或 Docling 这种强力解析器。
    *   **图像与表格剥离**：将图表单独切成块对象（Chunk），提取 Bounding Box 元数据，通过异步任务在后台交给 Qwen-VL 生成图表摘要。
*   **现实妥协（算力降平）**：
    *   搭建 **冷管道（Cold Pipeline）** 以异步任务队列（Celery 或 MQ）运行高保真解析。对用户而言展现为“文档解析中”的状态进度条，避免长时间阻塞 API 请求。
    *   容忍初期前 5% 的极端污损长难公式解析乱码，确保 95% 主体文字无损即可。

## 阶段 2：多维特征基质建立与混合路由检索 (Hybrid Indexing & Routing)
**理论对应**：相态 2：非线性检索场与异构图谱凝聚。
**现实约束**：构建全量 GraphRAG 极其昂贵（需要 LLM 遍历所有 Chunk 进行实体抽取，O(N^2) 复杂度），初创期很容易因为测试跑尽 API 余额。

*   **方向与动作**：
    *   **基础双路检索**：首先打通 Dense Vector（使用 Zhipu Embedding 或 BGE） + Sparse Vector（BM25 词频匹配）。
    *   **精准重排（Rerank）**：在这两路召回的结果上，架设一个交叉编码器（Cross-Encoder BGE-M3）进行重排序，确保关联度高的排在顶部。
    *   **渐进式图谱（Graph-Lite）**：暂不直接全量提取整本书的实体形成超大图谱。仅针对论文的“作者、机构、引用关系、核心关键词”建立轻量级元数据 Neo4j 拓扑。
*   **现实妥协（预算妥协）**：
    *   设计 **Query 降级路由（Fallback Router）**：当用户提问基础事实时，仅走便宜的“向量+重排”；只有当路由器识别出宏大综述（Macro-Summary）意图时，才高成本唤醒树状摘要或图谱社区查询。

## 阶段 3：受控生成与纠偏态反应堆 (Controlled Generation & CRAG/Self-RAG)
**理论对应**：相态 3：概率防浪堤与自我截断之门。
**现实约束**：真正的 Self-RAG 每生成一个词都需要额外的预测 Token 消耗，且生成速度极慢，用户体验在“秒回”时代可能会大打折扣。

*   **方向与动作**：
    *   **引入 LangGraph 状态机**：将大模型直接生成改为“状态流转”（Retrieve -> Grade/Critic -> Generate -> Reflect）。
    *   **幻觉纠偏（CRAG 机制）**：大模型生成前，先用一个小参数评分器检查“检索出的 evidence 能否支撑问题”。如果不能，直接终止并礼貌拒答，而不是瞎编。
*   **现实妥协（体验折中）**：
    *   实现 **模式分档**：为用户提供“深度思考模式”（完整执行多次重试、反思修正、证据链校验，允许等待 20-30 秒）与“极速问答模式”（牺牲严谨度，单次 RAG 问答，首字等待时间低于 2 秒）。
    *   强制模型生成统一附带标架引用的结构，要求输出如 `[Doc-1, P.4]` 这样的严格锚点，供前端悬浮溯源框提取。

## 阶段 4：定标护栏与白盒评估体系投入生产 (Observability & Golden Benchmarks)
**理论对应**：相态 4：标架无关的泛泛度量评估仪。
**现实约束**：盲目引入自动化评估框架（Ragas/TruLens）的打分有时比回答本身还不准不靠谱，且无法取代人类直觉。

*   **方向与动作**：
    *   **黄金测试集 (Golden Dataset)**：在项目初期不追求自动化打分系统，而是由开发组与一小批测试用户人工制作 100 个“典型刁钻学术问题+标准答案+来源锚点”组成验证集。
    *   **集成 LangSmith 或 Phoenix**：在后台录制每一次对话的实际检索耗时、Token 消耗、检索内容链路图，用于分析。
*   **现实妥协（可信度托底）**：
    *   自动化 CI/CD 测试仅用于回归防御（比如重排算法更新后，看在这 100 道题上是否出现准确率暴跌）。
    *   前端保留显眼的“人工复核 / Feedback（赞/踩/报错）”入口，让用户在使用中补齐真实标注数据，形成数据飞轮。

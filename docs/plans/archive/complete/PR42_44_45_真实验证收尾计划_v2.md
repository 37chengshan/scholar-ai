# 基于 PR42 / PR44 / PR45 的真实验证收尾计划

## 1. 目标

这次不再继续扩大架构改造范围。

本轮只做一件事：

> 基于已经合入的 PR42 / PR44 / PR45，验证检索主链在真实环境下是否真的可运行、可复跑、可比较。

输出要达到两个层级：

1. **小中规模真实检索验证闭环**：3 轮真实检索测试稳定跑完。
2. **大规模真实检索验证闭环**：至少 1 轮，建议 2 轮大型数据真实测试跑完，并给出 Milvus / Qdrant 对比结论。

---

## 2. 本轮不再做的事

- 不再重新设计 PR-A ～ PR-E 的工程方案。
- 不再返工页面层与 Notes / Chat / Read ownership。
- 不再新增大范围 retrieval 架构抽象。
- 不把本轮重点放在新增功能，而是放在**真实性验证与结果归档**。

一句话：

> 这轮是 **validation close-out**，不是 **engineering redesign**。

## 4. 收尾原则

### 原则 A：先验证主线 Milvus，再验证 Qdrant
主线仍然是 Milvus。

Qdrant 在本轮中属于：
- 同口径对照组
- 大数据实验组
- 迁移决策参考

### 原则 B：先跑小中规模闭环，再上大规模
先证明链路是对的，再证明规模能扛住。

### 原则 C：所有结论都要落到 artifacts
本轮所有结果必须输出到：
- JSON
- Markdown
- 可复跑命令记录

建议目录：
- `artifacts/benchmarks/real/`
- `artifacts/benchmarks/large-scale/`
- `docs/plans/archive/reports/retrieval-validation/`

## 4.1 你刚列的 7 个关键问题，已明确并入本轮计划

这 7 条不是附注，而是本轮验证收尾的**前置门禁**。执行顺序固定如下：

### Gate-1：统一 embedding 配置并保证真实检索能初始化

### Gate-2：Milvus / Milvus Lite 环境可用

直接检查文件：

### Gate-3：修正 run_search_benchmark.py 的 fixture 读取逻辑

退出条件：

## 5. 第一阶段：验证前收口

### 5.1 配置冻结
先冻结本轮验证用配置，不允许边跑边换。

#### 必查配置
- `VECTOR_STORE_BACKEND`
- `EMBEDDING_MODEL`
- `EMBEDDING_DIMENSION`
- `RERANKER_MODEL`
- `MILVUS_*`
- `QDRANT_*`
- `RETRIEVAL_TRACE_*`

#### 退出条件
- Milvus 配置可启动
- Qdrant 配置可启动
- embedding / reranker 与当前 collection 维度不冲突

### 5.2 数据集冻结
本轮至少准备两套数据：

#### Dataset-S（小中规模）
用途：3 轮稳定性验证

建议规模：
- 20–50 篇论文
- 3,000–10,000 chunks
- 覆盖 text / image / table / mixed query

#### Dataset-L（大规模）
用途：大型数据真实测试

建议规模：
- 300–1,000 篇论文
- 100,000–300,000 chunks

如果机器资源允许，再准备：

#### Dataset-XL（可选）
- 1,000+ 篇论文
- 500,000–1,000,000 chunks

#### 退出条件
- 每套数据都有 manifest
- 每套数据都有 `paper_id -> title -> source file` 映射
- 每套数据都可重建索引

### 5.3 查询集冻结
#### QuerySet-S
用于 3 轮稳定性测试：
- 30–50 条 query
- 覆盖 single / compare / multimodal / cross-paper

#### QuerySet-L
用于大规模测试：
- 50–100 条 query
- 增加 filter / topk / long-tail query

#### 退出条件
- query 集不再临时增删
- gold evidence 版本固定
- 每条 query 都能映射到目标 paper_ids 或 expected evidence

## 7. 第三阶段：1–2 轮大型数据真实测试

这是你额外要求的重点。

---

### Large Round A：Milvus 大型数据真实测试（必做）

#### 目标
验证主线 Milvus 在大规模 chunk 下的真实检索质量与延迟。

#### 使用数据
- Dataset-L
- QuerySet-L

#### 后端
- Milvus

#### 模式
- warm start 为主
- trace off
- reranker on

#### 必看指标
- Recall@5 / Recall@10
- MRR
- latency p50 / p95 / p99
- topk=20 / 50 / 100 对延迟的影响
- filter correctness
- reranker 后质量提升幅度

#### 建议分 3 组执行
1. dense only
2. dense + sparse
3. dense + sparse + rerank

#### 建议输出
- `large_milvus_dense.json`
- `large_milvus_hybrid.json`
- `large_milvus_hybrid_rerank.json`
- 对应 Markdown 汇总

#### 通过标准
- 全量 query 能跑完
- 指标可解释
- 没有大面积 timeout / OOM / collection 错配

---

### Large Round B：Qdrant 大型数据真实对比（强烈建议做）

#### 目标
在同一数据、同一 query、同一指标下，验证 Qdrant 是否值得继续投入。

#### 使用数据
- Dataset-L
- QuerySet-L

#### 后端
- Qdrant

#### 模式
- warm start
- trace off
- reranker on

#### 对比口径
必须与 Milvus 一致：
- 相同数据快照
- 相同 query 集
- 相同 embedding / reranker
- 相同 topk
- 相同 filter 条件

#### 必看指标
- Recall@k
- MRR
- latency p50 / p95 / p99
- ingest speed
- filter correctness
- 内存/磁盘占用
- 本地开发易用性

#### 通过标准
- 输出双后端可比较报告
- 至少能得出一个明确结论：
  - Milvus 明显更稳
  - Qdrant 明显更省资源
  - 或者二者差距不足以支撑迁移

## 8. 第四阶段：可选的重启后回归测试

### Reboot Regression Round（可选但推荐）

#### 目标
排除缓存、残留连接、旧 collection 状态造成的“伪稳定”。

#### 执行动作
- 完整停止服务
- 清理本轮临时缓存
- 重启 Milvus / Qdrant / API
- 再跑一轮 Dataset-L + QuerySet-L

#### 建议至少做一种
- Milvus reboot regression
- 若时间够，再加 Qdrant reboot regression

#### 验证点
- 服务重启后是否还能正确加载 provider
- collection/索引是否正确恢复
- 结果是否与 Large Round A/B 大体一致

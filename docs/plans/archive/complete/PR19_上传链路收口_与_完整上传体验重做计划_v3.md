# PR19：上传链路收口_与_完整上传体验重做计划

## 1. 背景与目标

本计划承接前一阶段的导入链路稳定性重构，默认 PR18 已完成以下方向：

- Import worker 去阻塞化
- Completion 双保险
- Dedupe 决策续跑闭环
- 批量 local_file 上传后端闭环
- 最小测试覆盖

PR19 的目标不是继续改 Celery 运行模型，而是做两件事：

1. **Worker 收口**  
   把当前 worker / task / callback / DB 访问边界进一步统一，降低后续维护复杂度与隐性风险。

2. **前端完整上传体验重做**  
   把上传从“能传文件”升级成真正可用的上传工作台，覆盖：
   - 断点续传
   - 秒传
   - 断网恢复
   - 批量上传
   - 任务恢复
   - 上传与 ImportJob 工作流统一

## 3. 核心设计原则

### D1. Upload 是一等工作流，不是对话框副功能
上传不再只是 `ImportDialog` 里的一个动作，而是独立的 Upload Workspace / Upload Session。

### D2. ImportJob 仍然是一等状态对象
上传体验不能绕开 ImportJob。上传、秒传、分片、断点恢复最终都要落回 ImportJob，避免前后端两套状态源。

### D3. Worker 只负责后台处理，不负责前端上传控制
前端上传负责：
- 选择文件
- 计算 hash（可选）
- 分片传输
- 状态恢复
- 用户交互

worker 负责：
- 接到“文件已就绪”的信号后开始处理
- 推进 ImportJob / ProcessingTask

### D4. 上传状态与解析状态分层
必须分开两类状态：
- Upload state：selecting / hashing / checking / uploading / paused / completed / failed
- Import state：queued / running / awaiting_user_action / completed / failed / cancelled

### D5. 前端必须具备刷新恢复能力
刷新页面后，至少应能恢复：
- 已存在的 ImportJob 列表
- 当前 upload session
- 已完成/未完成分片信息
- 可继续上传的任务

---

## 4. 整体架构思路

### 4.1 后端新增 Upload Session 协议层
在 ImportJob 之上增加一层“上传会话”概念，用于承接分片、秒传、恢复。

推荐模型：
- ImportBatch：一组导入任务
- ImportJob：单个文件/来源的导入状态
- UploadSession：单个文件的上传会话
- UploadPart：分片状态（可不单独建表，先存在 JSON 字段或 Redis，再视情况升级）

### 4.2 UploadSession 职责
- 记录 uploadId
- 对应 importJobId
- 文件名 / size / mime / sha256
- 当前已上传分片
- 是否完成合并
- 是否已触发 import enqueue

## 8. 验收标准

### 上传体验
- 单文件上传可稳定完成
- 批量上传可部分成功
- 网络中断后可继续
- 刷新页面后未完成任务可恢复
- 已存在文件可秒传，不重复上传

### 导入工作流
- 上传完成后 ImportJob 自动进入后台处理
- 处理完成后 papers 列表自动刷新
- 失败任务可清晰定位是 upload 失败还是 import 失败

### worker 线
- ImportJob / ProcessingTask / finalize 边界明确
- worker 不再承担前端上传协议责任
- DB 访问方式统一，不再散乱

## 9. 风险与控制

### 风险 1：上传协议复杂度上升
控制：
- 先做最小可靠版 chunk protocol
- 不一次性做太多高级优化

### 风险 2：hash 计算影响前端性能
控制：
- 使用 Web Worker 或分片计算
- 大文件哈希允许后台计算并显示进度

### 风险 3：秒传引入误判
控制：
- sha256 + size 双校验
- 服务端最终校验文件可用性

### 风险 4：恢复逻辑导致前后端状态不一致
控制：
- 后端 `GET upload-session` 作为真相源
- 前端本地状态只作缓存，不作权威状态

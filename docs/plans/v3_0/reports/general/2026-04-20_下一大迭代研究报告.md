# 1. 执行摘要

当前项目已经完成了一轮“基础设施、治理、页面壳层、前端视觉改造”的大动作，但还没有完全进入“用户任务闭环稳定可用”的阶段。

从计划台账、架构文档、4 月 18 日审查报告、以及当前代码实现交叉判断，ScholarAI 的下一大迭代最应该做的，不是继续追求局部 UI 美化，也不是立刻把重心切到更深的 hybrid retrieval / claim-level synthesis，而是把最核心的知识工作流打穿：

1. 用户上传或导入论文。
2. 文件进入可恢复、可继续的处理链路。
3. 论文进入知识库并可检索。
4. 用户在知识库上下文中提问。
5. 会话历史与流式消息保持一致并可恢复。

一句话概括：

**下一大迭代应定义为“知识工作流闭环迭代”，目标是把 ScholarAI 从页面级可用，推进到任务级可信。**

# 3. 当前阶段判断

## 3.1 已经完成的基础

这一轮不是“什么都没做”，相反，基础层已经很厚：

1. 治理真源已经建立。
   - [docs/plans/PLAN_STATUS.md](../plans/PLAN_STATUS.md) 已经把 active / done / superseded 关系拉清。
   - [docs/specs/governance/phase-delivery-ledger.md](../governance/phase-delivery-ledger.md) 已经定义交付单元与回填规则。
2. 架构边界已经明确。
   - [docs/specs/architecture/system-overview.md](../architecture/system-overview.md) 把 apps/web、apps/api、异步任务、SSE 责任边界固定下来了。
3. 前端工作台壳层已经成型。
   - Chat、KB、Search 都已经进入 features/workspace 结构，不再是单纯 page 直写。
4. 上传链路后端基础已经落地。
   - [apps/api/app/services/upload_session_service.py](../../apps/api/app/services/upload_session_service.py) 已具备 create session、register part、complete、abort、秒传复用等骨架。
   - Upload DTO 已进入共享类型与 SDK：[packages/types/src/kb/import.ts](../../packages/types/src/kb/import.ts)、[packages/sdk/src/kb/import.ts](../../packages/sdk/src/kb/import.ts)。
5. 会话历史与消息持久化基础已经存在。
   - [apps/api/app/services/message_service.py](../../apps/api/app/services/message_service.py) 已支持保存消息、分页读取、计数、更新。
   - [apps/api/app/api/session.py](../../apps/api/app/api/session.py) 已暴露会话消息读取接口。
6. Chat 前端视觉层刚完成一轮深改。
   - 这意味着下一轮不必再把主精力放在表层 UI 风格上。

## 3.2 当前最明显的缺口

问题不在“有没有组件”，而在“关键任务是否已经稳定闭环”。

### A. Chat 仍然存在大控制器

[apps/web/src/features/chat/workspace/ChatWorkspaceV2.tsx](../../apps/web/src/features/chat/workspace/ChatWorkspaceV2.tsx) 顶部明确写了 `LEGACY FREEZE (PR10)`，说明这个文件本身已经被认定为迁移中单体控制器，不应继续承接新业务逻辑。但它当前仍然同时处理：

1. session 搜索与过滤。
2. scope 解析与 URL 同步。
3. SSE 生命周期。
4. placeholder message 绑定。
5. 删除确认与右侧面板状态。

这说明 Chat 页面虽然“能跑”，但主干职责仍未真正下沉。

### B. KB 工作台已经有入口，但流程仍然是“多块功能拼装”

[apps/web/src/features/kb/components/KnowledgeWorkspaceShell.tsx](../../apps/web/src/features/kb/components/KnowledgeWorkspaceShell.tsx) 同时编排：

1. papers
2. import-status
3. uploads
4. search
5. runs
6. chat

而且导入完成后的刷新逻辑还是手工 fan-out：`loadImportJobs + loadPapers + loadKnowledgeBase + reloadRuns`。这类 callback 风格说明它还没形成真正的 workflow state machine。

### C. 上传链路“后端有基础，前端只完成第一层”

后端侧的 Upload Session 已经不是空壳，但前端恢复与传输控制仍偏薄：

1. [apps/web/src/features/uploads/hooks/useChunkUpload.ts](../../apps/web/src/features/uploads/hooks/useChunkUpload.ts) 当前是串行逐片上传，没有并发调度、失败重传、网络抖动策略。
2. [apps/web/src/features/uploads/hooks/useUploadRecovery.ts](../../apps/web/src/features/uploads/hooks/useUploadRecovery.ts) 只是对 `getSession` 的薄封装，还不等于真正的刷新恢复、断网恢复、本地状态回放。
3. [apps/web/src/features/uploads/hooks/useUploadWorkspace.ts](../../apps/web/src/features/uploads/hooks/useUploadWorkspace.ts) 负责队列状态，但恢复逻辑仍然停留在“拿服务端状态回写当前 item”这一层。

### D. 持久化与 SSE 有骨架，但还需要端到端闭环

1. [apps/api/app/services/message_service.py](../../apps/api/app/services/message_service.py) 证明消息落库路径已经存在。
2. [apps/api/app/api/session.py](../../apps/api/app/api/session.py) 证明历史读取接口已经存在。
3. [apps/api/app/api/chat.py](../../apps/api/app/api/chat.py) 说明 `message_id` 和 replay 语义已经被当作硬约束处理。

但这三者组成的只是“能力骨架”，不是“已被充分验证的端到端产品闭环”。

### E. Search 已经能用，但还不是这一轮的主矛盾

[apps/web/src/features/search/components/SearchWorkspace.tsx](../../apps/web/src/features/search/components/SearchWorkspace.tsx) 已具备 workspace 结构；[apps/web/src/features/search/hooks/useUnifiedSearch.ts](../../apps/web/src/features/search/hooks/useUnifiedSearch.ts) 目前只是对 `useSearch` 的轻封装，而不是独立复杂工作流。

这说明 Search 还有优化空间，但它更像“第二层提效问题”，不是当下最该优先打穿的主链路问题。

# 4. 当前 active 计划说明了什么

从 [docs/plans/PLAN_STATUS.md](../plans/PLAN_STATUS.md) 看，真正仍在 active 的主线是：

1. PR10：workspace 分层与稳定化。
2. PR19：上传链路收口与完整上传体验重做。
3. PR21-PR23：契约收口与持久化闭环。
4. PR7-PR8：Chat 稳定性与 RAG 能力升级。

这四条线其实不矛盾，它们共同指向一个更高层的产品目标：

**把 ScholarAI 的核心工作流从“多个子系统已存在”推进到“一个任务链路能稳定跑完”。**

也就是说，真正的下一大迭代，不应该被拆成四条彼此竞争的主题，而应该被重新命名成一个上层目标：

## 建议的大迭代主题

**知识工作流闭环迭代**

英文可以叫：

**Knowledge Workflow Closure**

# 5. 为什么不是别的方向

## 5.1 不是继续做纯 UI 深打磨

原因：

1. Chat 页刚完成深度视觉改造。
2. 当前最大风险不是“长得不够好”，而是“流程偶尔断、状态不统一、恢复不完整”。
3. 再做一轮纯视觉提升，不会提升 ScholarAI 的核心可信度。

## 5.2 不是立刻把重心切到更深的 RAG 升级

原因：

1. PR7-PR8 中的 hybrid retrieval、claim-level synthesis、citation verifier 确实重要。
2. 但它们建立在更稳定的上传、知识库、会话、SSE、持久化基础之上。
3. 如果工作流底盘还没收口，越往上叠高级 RAG，排障与验证成本越高。

## 5.3 也不是单独做治理自动化

原因：

1. PlanA / PlanB / PlanC 已经把治理脚本和台账铺起来了。
2. 治理当然还要继续补，但它更适合作为本轮大迭代的“约束系统”和“验收系统”，不应该反过来取代产品主线。

# 6. 候选大迭代主题对比

| 主题 | 用户价值 | 依赖解锁价值 | 工程杠杆 | 风险 | 结论 |
|---|---|---|---|---|---|
| 继续做 Chat 视觉深化 | 中 | 低 | 中 | 低 | 不适合作为主线 |
| 直接推进更深 RAG 能力 | 高 | 中 | 中 | 高 | 应作为下一阶段，不是当前第一优先级 |
| 只做治理自动化 | 低到中 | 高 | 高 | 低 | 适合作为并行保障，不适合作为产品主线 |
| 知识工作流闭环 | 很高 | 很高 | 很高 | 中 | **推荐作为下一大迭代主线** |

# 7. 我对“下一大迭代目标”的正式建议

## 7.1 目标名称

**ScholarAI 知识工作流闭环迭代**

## 7.2 迭代目标

在一轮集中迭代内，把以下用户任务做到真正稳定：

1. 新建知识库并导入第一篇论文。
2. 上传中断后恢复并继续处理。
3. 文件处理完成后能立即进入 KB 检索与 KB 问答。
4. Chat 会话历史可读回，流式消息不重复、不丢失、不漂移。
5. 从 Search 导入到 KB 的动作反馈清晰，不造成上下文跳失。

## 7.3 这一轮不追求什么

1. 不做大规模模型栈替换。
2. 不把重点放在“更炫的界面”。
3. 不把 hybrid retrieval / claim-level synthesis 作为第一优先级交付目标。

# 8. 推荐的三条执行主线

## 主线 A：Upload / Import 闭环硬化

目标：把 PR19 变成真正的用户可用能力，而不是“后端协议到位、前端先能跑”。

应做事项：

1. 给 [apps/web/src/features/uploads/hooks/useChunkUpload.ts](../../apps/web/src/features/uploads/hooks/useChunkUpload.ts) 增加并发上传、失败分片重试、超时与取消控制。
2. 给 [apps/web/src/features/uploads/hooks/useUploadRecovery.ts](../../apps/web/src/features/uploads/hooks/useUploadRecovery.ts) 补齐刷新恢复、本地队列恢复、断网恢复策略。
3. 让 Upload Workspace 与 KB workspace 的刷新逻辑从 callback fan-out 变成状态驱动。
4. 补齐 Upload Session 的端到端测试，包括秒传、缺失分片恢复、complete/abort。

为什么这条线是 P0：

因为 ScholarAI 的第一性价值来自“把论文变成可问答知识”，而不是先让用户看见一堆面板。

## 主线 B：契约、持久化、SSE 收口

目标：把 PR21-PR23 从“基础能力已存在”推进到“产品链路已验证”。

应做事项：

1. 锁定消息唯一写口，避免 assistant/tool message 再出现双写或脏数据。
2. 补齐 `chat/stream -> sessions/{id}/messages` 的集成回归。
3. 验证 [apps/api/app/api/chat.py](../../apps/api/app/api/chat.py) 的 replay、message_id、done/error 语义在前后端一致。
4. 同步回填 [docs/specs/architecture/api-contract.md](../architecture/api-contract.md) 与 [docs/specs/domain/resources.md](../domain/resources.md) 的真实状态。

为什么这条线是 P0：

因为没有稳定的契约与历史一致性，Chat 的所有高级能力都会变成高维护成本资产。

## 主线 C：围绕闭环做工作台收口，而不是再开新壳

目标：让 PR10 从“workspace 已经存在”走到“workspace 真正承接业务”。

应做事项：

1. 把 [apps/web/src/features/chat/workspace/ChatWorkspaceV2.tsx](../../apps/web/src/features/chat/workspace/ChatWorkspaceV2.tsx) 的 session / stream / send / scope 控制逻辑继续下沉，页面容器只做编排。
2. 把 [apps/web/src/features/kb/components/KnowledgeWorkspaceShell.tsx](../../apps/web/src/features/kb/components/KnowledgeWorkspaceShell.tsx) 的多路刷新改造成更明确的状态转换。
3. Search 只做与闭环直接相关的优化，例如导入反馈与返回 KB 的连续性；不在这一轮深挖右侧分析面板。

为什么这条线是 P1：

因为它决定本轮闭环做完后是不是还会继续回退到大组件补洞。

# 9. 建议的迭代边界

## 9.1 必做

1. Upload Session 前后端闭环与恢复。
2. ImportJob、Upload、KB 刷新与 run history 联动。
3. Chat stream 与 message persistence 端到端验证。
4. 与上述能力直接相关的工作台状态下沉。

## 9.2 可选但不应抢主线

1. Search 翻页继续提速。
2. Chat 右侧分析面板再优化。
3. 更深层次的检索与回答质量优化。

## 9.3 明确排除

1. 大规模重写后端任务系统。
2. 全量替换 Milvus / Celery / FastAPI 等底座。
3. 再做一轮以审美为中心的前端改版。

# 10. 3 周执行建议

## Week 1：把“基础能力存在”变成“链路已验证”

1. 补 Upload Session 后端集成测试。
2. 补 chat persistence 与 SSE 端到端回归。
3. 固化 contract 文档与验证命令。

交付结果：

1. Upload / Chat / Session 三条关键链路有明确自动化验证。
2. 文档与代码状态同步。

## Week 2：完成前端上传与 KB 工作台闭环

1. 实现上传队列恢复、失败重试、并发上传。
2. 打通 Upload Workspace -> Import Status -> Papers -> Runs 的状态联动。
3. 消除刷新后丢上下文或多处手工刷新不一致的问题。

交付结果：

1. 用户可以稳定完成上传和导入。
2. KB 页面不再靠多组 callback 拼刷新。

## Week 3：Chat 页面围绕闭环收口

1. 继续拆 ChatWorkspaceV2 控制职责。
2. 收束 session、scope、stream、placeholder 的边界。
3. 补四条北极星链路的 E2E / 页面级回归。

交付结果：

1. Chat 不再只是“视觉更像成品”，而是状态流更稳定。
2. 该轮迭代形成真正可发布的产品级闭环。

# 11. 关键风险与防控

## 风险 1：文档继续落后于代码

现在已经能看到这种迹象：旧审查结论里把某些能力判断为“未接线”，但代码里其实已实现，例如 Chat 会话搜索已经在 [apps/web/src/features/chat/workspace/ChatWorkspaceV2.tsx](../../apps/web/src/features/chat/workspace/ChatWorkspaceV2.tsx) 中完成过滤逻辑。

防控：

1. 这轮迭代每完成一条链路，都必须同步回填 [docs/plans/PLAN_STATUS.md](../plans/PLAN_STATUS.md)。
2. 所有 active 计划的 `last_verified_at` 和 `evidence_commits` 必须更新。

## 风险 2：范围再次发散成多条并行大工程

防控：

1. 只围绕“知识工作流闭环”收口。
2. Search 深度性能、RAG 高阶能力、治理自动化都不抢 P0 资源。

## 风险 3：只改 UI，不改状态真源

防控：

1. 任何前端改动都要优先落在 hook / store / workflow 层。
2. 不允许把新的复杂状态再塞回 [apps/web/src/features/chat/workspace/ChatWorkspaceV2.tsx](../../apps/web/src/features/chat/workspace/ChatWorkspaceV2.tsx) 这类冻结中的大控制器。

# 12. 最终判断

如果只能给一个结论，那就是：

**ScholarAI 的下一个大迭代，不该定义成“再优化 Chat”或“再升级 RAG”，而应该定义成“把 Upload -> Import -> KB -> Chat -> History 这一条知识工作流做成稳定、可恢复、可验证的闭环”。**

这是当前最能同时提升：

1. 用户感知价值。
2. 架构收口效率。
3. 后续 RAG 升级承载力。
4. 计划执行确定性。

# 1. 审查目标

本审查计划用于判断三份落实计划是否真正能够彻底解决审计报告中的全部问题，且可在工程上落地，不是纸面方案。

审查结论必须回答三件事：
1. 问题是否被完整覆盖。
2. 执行路径是否现实可行。
3. 风险与回滚是否足够可控。

# 2. 审查范围

纳入审查的问题来源：
1. 深度审计报告中的前端、后端、选型与里程碑问题。
2. Reverted Features 审计中关于 Phase 6A-8C 的落地缺口。
3. SSE/OCR 专项分析中的协议并存与默认策略问题。

# 3. 审查维度与评分

每个维度采用 0-5 分：
1. 覆盖性：是否覆盖全部高优先级问题。
2. 可执行性：任务粒度、依赖与工期是否可执行。
3. 可验证性：是否有量化验收与可观测指标。
4. 风险控制：是否提供明确回滚与止损机制。
5. 协同性：三计划之间是否存在冲突或遗漏。

评分阈值：
1. 任一维度低于 3 分：计划退回重写。
2. 平均分低于 4 分：计划不得进入执行。

# 4. 问题-计划覆盖追踪矩阵

## 4.1 前端问题映射

1. ChatLegacy 过重：计划 A 第 1-2 周拆分 + 第 6 周下线。
2. SSE 双轨并存：计划 A 第 3 周统一协议。
3. KB Detail 耦合：计划 A 第 4 周状态下沉。
4. UI 交互不一致：计划 A 第 5 周统一收口。

## 4.2 后端问题映射

1. 环境不稳定：计划 B 第 0 周标准化。
2. 启动重负载：计划 B 第 1-2 周链路重构。
3. OCR/解析闭环不足：计划 B 第 3 周补基准与证据。
4. retrieval fallback：计划 B 第 4-5 周强切清理。
5. confidence 旧逻辑：计划 B 第 6 周重算。
6. hybrid 与 claim-level 缺失：计划 B 第 7-8 周补齐。

## 4.3 治理问题映射

1. 规划与实现偏差：计划 C 第 1-2 周追踪框架。
2. 分支孤岛：计划 C 第 3-4 周分支治理。
3. 契约到期不退役：计划 C 第 5-6 周契约门禁。
4. 双端协同缺失：计划 C 第 7-8 周 E2E 阻断。

# 5. 审查流程（四轮）

## 第一轮：文档静态审查

检查项：
1. 每个问题至少被一个计划命中。
2. 每个计划都有阶段、里程碑、验收、回滚。
3. 不允许“后续再补”式空白描述。

输出：静态问题清单。

## 第二轮：依赖冲突审查

检查项：
1. 计划 A 与计划 B 的接口契约是否一致。
2. 计划 B 与计划 C 的门禁与上线节奏是否一致。
3. 是否存在资源同一时间超分配。

输出：依赖冲突图与修订建议。

## 第三轮：可执行性压测

检查项：
1. 将每周任务映射到人力与工时。
2. 抽样验证关键任务是否可在一周内完成。
3. 检查是否有过大批次上线风险。

输出：执行排期修正表。

## 第四轮：验收与回滚演练

检查项：
1. 验收指标是否能被实际采集。
2. 回滚链路是否可在规定时间完成。
3. 是否存在“不可回滚”的高风险步骤。

输出：上线前通过/阻断结论。

# 6. 审查门禁（必须全部满足）

1. 全问题覆盖率 100%。
2. 高风险任务全部具备回滚路径。
3. 关键指标可自动采集，不依赖手工拼接。
4. 三计划无互相阻断的硬冲突。
5. 审查意见关闭率达到 100%。

# 7. 审查角色与职责

1. 审查负责人：工程经理，负责裁决。
2. 前端审查人：负责计划 A 执行真实性。
3. 后端审查人：负责计划 B 契约与稳定性。
4. 平台审查人：负责计划 C 门禁与流水线。
5. QA 审查人：负责验收可测性与回归覆盖。

# 8. 审查节奏

1. 周一：提交本周修订版计划。
2. 周二：静态与依赖审查。
3. 周三：可执行性压测。
4. 周四：回滚演练。
5. 周五：发布审查结论与下周准入条件。

# 9. 阻断条件（任一命中即停止执行）

1. 关键问题无归属计划。
2. 验收指标不可测。
3. 高风险任务无回滚。
4. 分支治理与契约治理未落地却进入大规模重构。

# 10. 最终审查输出模板

每周输出必须包含：
1. 覆盖率分数卡。
2. 风险项清单与关闭进度。
3. 资源与排期偏差。
4. 本周准入/阻断结论。

# 11. 结论标准

只有在以下情况下，才判定三份计划“能彻底解决报告中的所有问题”：
1. 所有问题均有明确任务、负责人、时间窗、验收指标。
2. 没有依赖悬空项。
3. 没有未定义的回滚路径。
4. 已形成可持续治理机制，防止同类问题复发。

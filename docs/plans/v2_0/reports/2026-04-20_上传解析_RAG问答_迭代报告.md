# 1. 执行摘要

本报告聚焦 ScholarAI 当前最关键的产品主链路：

Upload/Import -> Parse/Index -> Knowledge Base -> RAG Query -> Citation/History

结论：

1. 当前架构已经具备 ImportJob 状态机、分片上传会话、外部源解析适配器和 RAG 查询能力，基础并不薄弱。
2. 真正瓶颈不在“有没有功能”，而在“是否形成稳定、可观测、可恢复、可扩展的任务闭环”。
3. 下一迭代建议优先做“链路可靠性与导入成功率”而非继续堆叠表层体验或高阶算法特性。

建议把下一迭代主标题明确为：

知识导入与问答闭环稳定化（Reliability First）

# 3. 当前系统能力基线

## 3.1 已具备的关键能力

1. ImportJob-first：先建作业再建 Paper，状态机路径明确。
2. 可恢复上传：upload session 支持分片、missingParts、complete/abort。
3. 外部源适配器：arXiv/S2/DOI/PDF URL 分别实现解析、元数据、下载逻辑。
4. 去重机制：外部ID、哈希、模糊标题三级去重。
5. 异步处理链路：导入 worker 与处理 task 解耦，避免主 worker 阻塞。
6. RAG 查询接口：支持检索源规范化与置信度解释字段。

## 3.2 当前闭环中的主要短板

1. 前端导入完成检测仍以轮询为主，且超时时间短（约 60 秒窗口），长处理任务体验不稳定。
2. 分片上传为串行策略，网络抖动或大文件场景吞吐偏低。
3. 前端分片前会对整个文件做 SHA-256，超大文件时对内存与主线程压力较大。
4. 后端 `/import-jobs/{job_id}/file` 采用整文件 `read` 到内存，不利于大文件扩展。
5. 虽有 SSE 导入进度流接口，但前端导入主流程尚未默认接入实时流。
6. DOI 路径的 PDF 获取策略对 `openAccessPdf` 依赖较强，缺少多源 OA 回退策略。
7. `pdf_url` 适配器严格依赖 HEAD + Content-Type，部分站点会误判（不支持 HEAD 或返回 octet-stream）。
8. 导入链路已有阶段信息，但端到端观测指标（队列耗时、阶段 SLA、失败分型）尚未产品化呈现。
9. RAG 置信度侧重检索证据强度，缺少“回答-证据一致性”独立评分。

# 4. 开源对标结论（对 ScholarAI 最有价值的部分）

## 4.1 RAGFlow

可借鉴点：

1. 文档解析与 chunking 管道明确分层，强调复杂文档理解。
2. 强调可解释引用与可追溯证据展示。

对 ScholarAI 的启发：

优先把“导入成功率 + 引用可追溯”做成核心质量指标，不先追求模型花样。

## 4.2 Onyx

可借鉴点：

1. 连接器同步与后台 worker 体系成熟。
2. 标准版架构显式包含队列、缓存、对象存储，强调生产化可靠性。

启发：

ScholarAI 需要从“功能可用”升级到“长任务可观测可恢复”。

## 4.3 Dify

可借鉴点：

1. RAG 管道从 ingestion 到 retrieval 的产品化路径清晰。
2. 观察性（LLMOps）与数据闭环强调持续优化。

启发：

导入阶段应形成结构化指标看板，不仅靠日志排障。

## 4.4 AnythingLLM

可借鉴点：

1. collector 独立服务承担文档解析处理。
2. 强调文档导入与聊天体验的一体化。

启发：

解析任务与问答请求应继续隔离负载，避免高峰互相影响。

## 4.5 Docling / GROBID

可借鉴点：

1. Docling 在多格式解析与结构化导出方面能力强，适合作为复杂 PDF/图表增强解析层。
2. GROBID 在学术文献元数据、参考文献、章节结构提取上有长期验证。

启发：

ScholarAI 可采用“默认轻量解析 + 失败或复杂样本回退增强解析”的双引擎策略。

# 5. 迭代策略：三阶段执行

## 阶段 A（2 周）：可靠性收口

目标：把导入-上传-完成主链路先做稳定。

任务：

1. 导入主流程接入 SSE（优先）+ 轮询兜底双通道。
2. 将导入状态机在前端显式化：created/queued/running/awaiting_user_action/completed/failed。
3. 上传失败重试策略分层：单分片重试、会话恢复、作业重试。
4. 统一错误码映射到前端可理解文案（网络、限流、无 OA PDF、格式非法、去重等待）。

验收指标：

- 导入任务状态可见率 100%。
- 导入状态“无反馈超时”问题下降到 < 1%。
- 用户可恢复上传成功率 > 98%。

## 阶段 B（3 周）：吞吐与鲁棒性增强

目标：在大文件和高并发场景下提高成功率与效率。

任务：

1. 分片并发上传（可配置并发度 2~4），加入动态退避。
2. 服务端文件写入由整文件内存读改为流式处理。
3. `pdf_url` 适配器增加 GET 回退校验（HEAD 失败时）。
4. DOI 下载路径新增 OA 回退源（Unpaywall/OpenAlex）。
5. 阶段级别 SLA 监控：resolving/fetching/downloading/processing 各阶段耗时分布。

验收指标：

- 50MB 文件导入成功率 > 99%。
- 大文件上传 p95 用时下降 20% 以上。
- 因外部源不可达导致的可恢复失败率下降 30% 以上。

## 阶段 C（2 周）：问答质量与证据可信度提升

目标：把“能回答”升级到“回答可置信、可解释”。

任务：

1. 在现有置信度外新增“答案-证据一致性分”。
2. 统一 citation 渲染协议（paper_id/section/page/anchor_text/source_id）。
3. 增加失败回答类型标注：证据不足、证据冲突、高不确定性。
4. 建立最小评估集：10~20 条真实查询，回归跟踪准确率与引用覆盖。

验收指标：

- 带引用回答占比 > 95%。
- 引用定位正确率 > 90%。
- 低置信回答显式提示率 100%。

# 6. 风险矩阵与应对

1. 外部 API 波动与限流
- 风险：导入等待时间波动大，失败率上升。
- 对策：指数退避、缓存、降级源切换、前端提示“延迟处理中”。

2. 开放 PDF 不可得
- 风险：DOI/S2 元数据存在但无法抓取 PDF。
- 对策：多源回退 + 手动上传接力流程 + 保留已解析元数据。

3. 前后端契约漂移
- 风险：状态字段、source 字段不一致导致流程中断。
- 对策：契约快照测试 + SDK 类型对齐 + 关键接口变更门禁。

4. 任务堆积
- 风险：高峰时处理任务排队导致用户误判失败。
- 对策：展示 queue ETA、提供后台通知、分级优先级。

# 7. 里程碑计划（建议）

- M1（第 2 周末）：SSE 导入进度接入 + 状态机统一展示完成。
- M2（第 5 周末）：并发分片、流式写入、OA 多源回退上线。
- M3（第 7 周末）：RAG 证据一致性评分与评估看板上线。

# 9. 多维度复审（已执行）与修订记录

复审维度：

1. 技术正确性
2. 可执行性（排期与依赖）
3. 风险覆盖
4. 指标可量化程度
5. 与现有代码一致性
6. 外部证据可追溯性

复审发现与已修订项：

1. 发现：初稿仅描述“建议接入 SSE”，未明确后端已具备 `/api/v1/import-jobs/{job_id}/stream`。
- 修订：补充“后端已有、前端未默认接入”的差距描述，避免误判为后端缺失。

2. 发现：阶段 B 的“大文件优化”表述偏泛。
- 修订：明确为“前端全文件 SHA-256 主线程压力 + 后端整文件内存读取”两个具体瓶颈。

3. 发现：错误处理建议没有对齐现有 ImportJob 状态与 next_action 设计。
- 修订：将建议改为“统一错误码映射 + 状态机显式化”，保持与现有 created/queued/running/awaiting_user_action/completed/failed 一致。

4. 发现：指标项存在“可感知但不易核验”风险。
- 修订：将关键指标改成可采集口径（状态可见率、恢复成功率、p95 用时、失败分型占比）。

5. 发现：RAG 提升建议中“一致性评分”缺少落地边界。
- 修订：限定为“在现有 confidence 基础上新增单独维度”，避免误读为全链路重构。

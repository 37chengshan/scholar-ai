# 05 Phase1 执行文档：v2.0 基线收敛与稳定性治理

## 1. Phase1 目标

在不引入新功能的前提下，完成 v2.0 的工程稳定性收敛：

1. 收敛后端全量 unit 存量失败
2. 固化前后端 release gate 验证流水线
3. 形成可重复、可审计、可回归的提交前验收流程

## 2. Phase1 入口条件

以下条件全部满足才可启动 Phase1 feature work：

1. Phase0 文档完整落地（5 份文档）
2. chat fast path 单测通过
3. chat critical/evidence/notes/responsive 全通过
4. session ownership 风险已修复并有测试覆盖

## 3. Phase1 工作包

### WP1: Backend Unit 失败分层

- 输出: 失败清单分层矩阵（环境依赖、测试漂移、真实缺陷）
- 验收: 每个失败项有 owner、修复路径、预估工时

### WP2: 失败簇优先级治理

- 优先处理:
  1) 会影响线上安全/数据隔离的失败
  2) 会影响 release gate 关键路径的失败
  3) 纯测试夹具或 mock 漂移问题
- 验收: 每轮至少关闭一个失败簇，单轮不引入新簇

### WP3: 测试环境基线固化

- 固化 Python/Node 版本与关键依赖约束
- 固化 E2E 账号与 auth state 生命周期
- 验收: 同一机器重复执行结果波动可控

### WP4: 验证脚本编排

- 统一入口:
  - 前端 type-check + 4 个关键 E2E
  - 后端 fast path + 分层 unit 组合
- 验收: 单命令可执行，结果可机读

## 4. Phase1 验收标准

1. 后端 unit 失败数显著下降并有基线报告
2. Phase0 P0/P1 场景无回归
3. 提交前验证流程在本地与 CI 一致
4. release gate 报告可由真实命令结果反推

## 5. 风险与回退

- 若单轮修复导致关键 gate 回归:
  1) 立即回退该轮变更
  2) 记录根因与最小复现
  3) 拆分为更小修复批次重试

## 6. 建议执行顺序

1. 先清理 ImportError/环境耦合类失败
2. 再清理测试夹具与接口漂移类失败
3. 最后做契约级回归和文档更新

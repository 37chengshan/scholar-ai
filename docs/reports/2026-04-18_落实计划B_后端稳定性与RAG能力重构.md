# 1. 计划目标

本计划用于一次性解决后端“可运行性差、契约并存、检索质量不可解释”三大根问题。核心策略是：
1. 运行时环境标准化。
2. 模型与依赖加载分层化。
3. RAG 契约强制统一化。
4. 质量指标可观测化。

# 2. 问题闭环清单

必须彻底关闭的问题：
1. 本地运行强依赖人工环境，解释器与依赖易漂移。
2. 启动阶段模型加载过重，导致启动失败定位困难。
3. OCR 与解析策略虽有恢复，但缺少性能基准与运行证据闭环。
4. retrieval contract 仍保留 fallback，导致双字段长期并存。
5. confidence 仍使用旧逻辑，结果可信度解释不足。
6. hybrid retrieval 与 claim-level synthesis 缺失，复杂问答质量受限。

# 3. 重构总设计

## 3.1 运行配置三档

1. dev-lite：默认本地开发档，最小依赖、模型延迟加载。
2. dev-full：联调档，启用全链路但允许降级。
3. prod：生产档，严格依赖、严格健康探针与告警。

## 3.2 启动链路分层

1. Preflight 层：版本、关键包、连接可达性自检。
2. Core API 层：先启动可用接口，不阻塞在模型加载。
3. AI Workers 层：embedding/reranker 后台就绪与健康探针。

## 3.3 RAG 契约单一真相

统一字段仅保留：
1. text
2. score
3. page_num
4. source_id
5. section_path
6. content_subtype
7. anchor_text

所有 similarity/page/content_data fallback 必须删除，不再容忍无限兼容。

# 4. 分阶段执行（8 周）

## 第 0 周：运行时标准化

动作：
1. 固化 Python 3.11 作为唯一受支持运行时。
2. 产出锁定依赖清单并建立可复现安装路径。
3. 增加一键 preflight 命令，启动前失败即中断。

产出：
1. 标准运行脚本。
2. preflight 报告模板。

## 第 1-2 周：启动链路重构

动作：
1. 将 lifespan 中的模型 eager-load 改为可配置策略。
2. 引入模型加载任务队列与就绪状态缓存。
3. 将 health 细分为 basic/ready/deep 三层探针。

验收：
1. 无模型情况下 basic health 可返回。
2. 模型异常不会拖死主进程。

## 第 3 周：解析与 OCR 策略闭环

动作：
1. 维持 do_ocr 默认 False，严格执行 native -> smart fallback。
2. 为 adaptive chunk size 增加边界测试与压测数据集。
3. 输出 born-digital 与扫描件性能基线报告。

验收：
1. born-digital 平均时延达到目标区间。
2. 扫描件 fallback 成功率达标。

## 第 4-5 周：retrieval contract 强制统一

动作：
1. 清理 agentic_retrieval 中 score/similarity/page/page_num 双轨 fallback。
2. 对输入数据增加契约断言和错误上报。
3. source 端同步升级，保证统一字段必填。

验收：
1. 新契约覆盖率 100%。
2. 无 fallback 代码残留。

## 第 6 周：confidence 重构

动作：
1. 将 confidence 重算为多因素模型：覆盖率、证据多样性、答案支撑度。
2. 记录每个因子权重与最终分值。
3. 在 API 返回中暴露 explain 字段供前端展示。

验收：
1. 相比旧算法，低质量答案的置信度显著下降。
2. 可解释信息可被前端直接消费。

## 第 7-8 周：hybrid retrieval 与 claim-level synthesis

动作：
1. 引入 dense + sparse 混合检索管线。
2. 增加 query planner 做多路查询分解。
3. 将答案生成从线性拼接升级为 claim-level 证据合成。
4. 引入 citation verifier 校验证据支撑。

验收：
1. 复杂问题答复完整度显著提升。
2. 引用可追溯率和一致率达标。

# 5. 数据与模型迁移策略

1. Schema 先扩后收：先新增新字段，再切写入，最后移除旧字段。
2. 读路径双读单写：迁移期允许读取旧字段但写入只写新字段。
3. 回填任务：历史索引分批补齐 section_path/content_subtype/anchor_text。
4. 切换策略：按租户或知识库维度分批切换。

# 6. 风险与回滚

## 6.1 关键风险

1. 契约强切导致历史数据读取失败。
2. 模型异步加载造成首请求体验波动。
3. 混合检索引入额外延迟。

## 6.2 回滚策略

1. 保留旧检索链路开关，但只在紧急回退启用。
2. confidence 算法保留旧版并支持版本化回退。
3. 每周上线仅一个高风险主题，失败即回切。

# 7. 验收矩阵

## 7.1 稳定性

1. 冷启动成功率 >= 99%。
2. 启动失败可定位率 100%。
3. 运行时崩溃率连续两周低于阈值。

## 7.2 契约一致性

1. 新契约字段完整率 100%。
2. 回归测试中旧字段引用次数为 0。
3. 契约断言失败均进入告警面板。

## 7.3 检索质量

1. Top-K 命中率提升目标达成。
2. claim 级引用支撑率达标。
3. confidence 与人工评审相关性提升。

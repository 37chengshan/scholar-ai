# 1. 这份 roadmap 解决什么

这份 roadmap 不再聚焦“前端能不能跑”，而是聚焦“用户能不能稳定、自然、可信地完成关键任务”。

当前前端的主要问题，不是页面缺失，而是以下三类体验债务同时存在：

1. 关键任务链路仍有中断点。
   - 例：看起来可用但实际未接线的能力、翻页闪烁、批量能力半完成、跨页面状态反馈不一致。
2. 高价值页面已具备骨架，但复杂度仍集中在单个大组件。
   - 例：`ChatWorkspaceV2.tsx` 仍承载过多职责；KB workspace 任务过多、分区不够清晰。
3. 前端对“稳定感”的塑造不足。
   - 例：翻页会整块 loading、错误恢复路径不统一、移动端退化策略不够明确。

目标不是再做一轮局部修补，而是把前端升级为“以任务为中心、以状态反馈为核心、以稳定感为竞争力”的产品界面。

# 2. 当前诊断

## 2.1 用户最容易感知的问题

### Chat

1. 对话主流程已经迁到 `ChatWorkspaceV2`，但该文件仍接近千行，混合了会话、作用域、SSE、placeholder、侧边栏和完成态逻辑。
2. 会话侧边栏搜索框已展示，但当前没有真实筛选逻辑接线，容易造成“看起来能用、实际没反应”的失信体验。
3. 思考、工具、引用面板已具备，但仍偏工程内部视角，尚未完全转成用户友好的“可理解过程”。

### Knowledge Base

1. `KnowledgeWorkspaceShell` 已经能承载 papers/import/uploads/search/runs/chat，但单页承载任务过多，认知负担偏高。
2. 用户在“导入、处理、检索、提问”之间跳转时，流程感仍不足，更像功能集合，不像工作流。

### Search

1. 搜索页已形成三栏结构，但结果翻页仍容易给人“重新搜索”的感觉。
2. 当前翻页会触发整块结果区 loading 替换，缺少保留上一页、预取下一页、局部更新等优化。
3. 右侧 analysis 区块部分内容更接近演示性信息，而非决策性信息。

### Global UI

1. loading、empty、error、retry、toast 已有部分统一组件，但实际页面还在混用多套反馈风格。
2. 导航、移动端、表单反馈、危险操作确认等体验仍有细节不一致。

## 2.2 代码层面的体验信号

1. 大体量文件仍集中在体验核心路径。
   - `apps/web/src/features/chat/workspace/ChatWorkspaceV2.tsx`
   - `apps/web/src/app/hooks/useChatStream.ts`
   - `apps/web/src/services/sseService.ts`
2. 搜索分页链路仍是裸 `useEffect + setLoading(true) + 请求完成后整体替换结果` 的模型。
3. 搜索翻页尚未使用 query cache、keep previous data、预取下一页、请求取消。
4. 若干“半能力”仍直接暴露给用户，例如 KB 列表里的 batch export 提示“暂未完全实现”。

# 3. 北极星体验目标

下一阶段前端优化应围绕 4 条北极星任务链路展开。

## 3.1 北极星任务链路

1. 创建知识库并成功导入第一篇论文
2. 从外部搜索发现论文并导入到指定知识库
3. 在知识库内发起提问并基于引用继续追问
4. 回到历史会话并在原上下文中继续工作

## 3.2 北极星体验指标

1. 用户点击后 100ms 内必须看到明确反馈
2. 关键路径中不出现“按钮可点但没有真实行为”的伪交互
3. 翻页、切 tab、切 session 时不出现整页闪烁
4. 错误态必须给出恢复动作，而不是只给错误文本
5. 移动端至少保证“可完成核心任务”，即使不是完整桌面体验

# 4. Roadmap 总览

建议按 8 周推进，分为 4 个阶段。

## Phase 0（第 0-1 周）：体验基线与止损

目标：先清除最伤信任感的问题，再进入结构优化。

### 产出

1. 前端体验基线文档
2. 假能力清单与处理策略
3. 统一状态反馈规范

### 必做事项

1. 下线或隐藏未真正完成的前端入口。
   - 例如 KB batch export 这类仍是 stub 的能力，未完成前不要直接面向用户暴露。
2. 清理“输入框看似可用但未接线”的交互。
   - 例如 chat session sidebar 的搜索框。
3. 梳理全局 loading、empty、error、retry、toast 的语义分层。
4. 明确 destructive action 统一规范。
   - 删除知识库、删除 session、取消上传、停止生成必须有一致的确认与结果反馈。

### 验收

1. 不再出现用户可见的“点击后只提示未实现”
2. 核心页面状态反馈样式和措辞统一

## Phase 1（第 1-3 周）：Chat 旗舰体验打磨

目标：把 chat 从“技术上可流式工作”升级成“用户愿意连续使用”的主入口。

### 重点方向

1. 降低 `ChatWorkspaceV2` 复杂度
   - 把 session 管理、scope 管理、stream lifecycle、message feed、agent side panels 进一步下沉到 hook/store 层
   - 页面容器只负责编排，不直接处理细粒度状态拼接
2. 会话列表体验升级
   - 真正支持搜索
   - 支持按最近活跃排序
   - 对空会话、长会话、已删除会话给出明确表现
3. 消息流体验升级
   - 发送后立即给出稳定 placeholder
   - done/error/cancel 后状态收束明确
   - 停止生成、继续追问、错误重试形成闭环
4. 右侧过程面板升级
   - 从“技术 telemetry”转为“用户可理解的过程说明”
   - 默认只暴露最重要信息，深入信息按需展开
5. scope 感知升级
   - 单篇论文 / 整个知识库 / 通用对话三种范围的状态更加可见
   - 进入作用域、退出作用域、当前上下文边界要更明确

### 验收

1. Chat 全流程中没有未接线控件
2. 发送、停止、报错、完成后的反馈一致且可恢复
3. 首次发消息到看见 assistant placeholder < 100ms

## Phase 2（第 3-5 周）：Knowledge Base 工作台重组

目标：把 KB 页面从“标签页集合”升级成“清晰工作流工作台”。

### 建议结构

1. Overview
   - KB 基本信息
   - 最近导入
   - 最近问答
   - 最近失败
2. Ingest
   - 本地上传
   - 外部导入
   - 导入队列
   - 失败修复
3. Use
   - 知识库检索
   - 快速提问
   - run 历史

### 重点方向

1. 减少同一层级上同时出现过多任务入口
2. 让“导入 -> 等待处理 -> 检索 -> 提问”变成自然的连续流
3. 明确当前 KB 的状态
   - 空库
   - 导入中
   - 可检索
   - 部分失败

### 验收

1. 新用户进入 KB 详情页时，能在 5 秒内理解下一步该做什么
2. 与导入相关的操作不再分散在多个互不连贯的入口中

## Phase 3（第 5-7 周）：Search 发现体验与分页性能专项

目标：让 Search 成为“发现 -> 判断 -> 导入 -> 阅读 -> 提问”的高效入口，而不是一个会抖动的结果页。

### 重点方向

1. 搜索结果翻页不卡
2. 搜索状态与筛选切换稳定
3. 减少演示性信息，增加决策性信息
4. 打通导入、阅读、问答的后续动作

### 搜索分页不卡专项

当前推荐的专项改造顺序：

1. 区分 `initialLoading` 和 `pageFetching`
   - 首次搜索时允许全区 loading
   - 翻页时保留上一页结果，不整块清空
2. 引入 query cache
   - 以 `query + filters + page` 为 key
   - 命中过往页时秒开
3. 开启 `keepPreviousData`
   - 翻页时保留上一页结果，局部显示加载条或按钮 loading
4. 预取下一页
   - 当前页到达后，在后台预取下一页
5. 请求取消与乱序保护
   - 用户快速切 query / sort / page 时，只接受当前上下文的最后响应
6. 局部重渲染
   - `SearchResultsPanel`、`SearchPagination`、右侧面板各自 memo 化
   - 翻页时不让整页动画重新播放
7. 滚动与焦点恢复
   - 翻页后滚动定位到结果区顶部
   - 保持焦点不跳回全页顶部

### 不建议优先做的事项

1. 不建议一开始就上虚拟滚动
   - 当前 `PAGE_SIZE = 20`，卡顿更大概率来自 loading 替换与无缓存，而不是节点过多
2. 不建议继续增加右侧静态分析块
   - 应先把结果区体验和导入闭环打稳

### 验收

1. 翻页时结果区不闪白、不整块消失
2. 已访问页再次进入时响应接近瞬时
3. 连续翻页时无旧请求覆盖新结果的抖动问题

## Phase 4（第 7-8 周）：全局一致性、移动端、无障碍、质量门禁

目标：把零散体验优化收口到产品级标准。

### 重点方向

1. Layout 细节修正
   - 顶栏、移动菜单、用户头像区、设置/登出行为统一
2. 移动端退化策略
   - Search 三栏在窄屏上的退化方案
   - Chat 侧边栏与右侧面板在窄屏上的切换方案
   - KB workspace 在手机和小平板上的主次层级重排
3. Accessibility
   - 焦点顺序
   - 键盘导航
   - 错误提示可被读屏识别
   - 颜色与对比度检查
4. 回归门禁
   - Playwright 核心任务流
   - 截图基线
   - 控制台告警预算

### 验收

1. 核心路径具备桌面 + 窄屏可用性
2. 控制台噪音显著下降
3. 关键流程有稳定回归保护

# 5. 按工作流拆分的优化主题

## 5.1 Workflow A：首次创建 KB 并导入论文

当前风险：

1. 入口较多，导入方式与上传方式并列，用户可能不理解区别
2. 导入成功后的下一步引导不够强

优化方向：

1. 首次进入 KB 时突出“上传第一篇 PDF”主路径
2. 对外部导入、本地上传给出更清晰的场景说明
3. 导入成功后提供明确的 next step
   - 继续导入
   - 查看导入状态
   - 进入检索
   - 开始提问

## 5.2 Workflow B：从 Search 发现并导入论文

当前风险：

1. 搜索页重点分散
2. 导入动作与结果页之间的关系还不够紧密

优化方向：

1. 结果卡片上突出“为什么值得导入”
2. 导入后给出明确反馈和后续入口
3. 已导入 / 可导入 / 导入中 3 种状态在卡片上可区分

## 5.3 Workflow C：在 KB 内提问并继续追问

当前风险：

1. scope 状态虽然存在，但用户未必理解
2. 完成后“继续追问”的引导不够强

优化方向：

1. 让当前提问范围始终可见
2. 引用点击后能自然进入阅读页并可回到原对话
3. 会话完成后提供清晰的 follow-up affordance

## 5.4 Workflow D：回到历史会话继续工作

当前风险：

1. session 列表搜索未接线
2. 长会话定位能力不足

优化方向：

1. 支持 session 搜索
2. 支持按最近活跃和标题快速筛选
3. 会话切换后保持更稳定的滚动和选中态

# 7. 量化指标

## 7.1 体验指标

1. 点击到首个视觉反馈 < 100ms
2. Search 翻页切换到可交互状态 < 300ms（缓存命中场景）
3. Search 新页首屏可见内容替换 < 800ms（非缓存场景，正常网络）
4. Chat 发送到 placeholder 出现 < 100ms
5. 导入成功后用户进入下一步的点击转化率明显提升

## 7.2 稳定性指标

1. 控制台 warning/error 日均下降
2. Search 翻页错误率接近 0
3. Chat 流式会话中断后可恢复率 > 95%

## 7.3 维护性指标

1. `ChatWorkspaceV2` 继续降到更合理的体量
2. Search 数据流从手写 effect 迁移到 query cache 模型
3. 页面内直接处理复杂异步逻辑的代码量下降

# 8. 风险与取舍

## 8.1 最大风险

1. 一边做视觉 polish，一边继续堆功能，会把体验工程重新打散
2. Search 若只改 UI，不改数据流，翻页和筛选的“卡”和“抖”不会真正消失
3. KB 页面若只是继续加 tab，会让工作台更强但更难用

## 8.2 取舍建议

1. 先做稳定感，再做装饰性增强
2. 先打通 4 条北极星任务链路，再扩展边缘能力
3. 先减少假能力，再追求看起来“功能很多”

# 9. 推荐执行顺序

1. 前序 API/SSE/persistence 计划完成
2. 体验基线与假能力清理
3. Search 分页不卡专项
4. Chat 体验硬化
5. KB 工作台重组
6. Search 转化链路优化
7. 移动端/无障碍/回归门禁

# 10. 结论

如果前序契约和持久化问题都已收口，前端下一阶段最值得投入的，不是“再补一个页面”，而是把用户的关键工作流做顺。

真正能拉开差距的三件事是：

1. Chat 变得稳定、可信、可持续使用
2. KB 工作台从功能集合变成真正的研究工作台
3. Search 翻页和导入体验从“能用”变成“丝滑且高转化”

这三件事做好后，前端才算从“工程迁移完成”进入“产品体验可持续优化”的阶段。

# 11. 本次 PR 审阅结论（稳定可用优先）

基于当前代码实况，本次 PR 不做“全量重构”，而是先完成“稳定可用”的闭环，优先修复会直接伤害信任感与可用性的点。

## 11.1 已确认的高优问题

1. Chat 会话侧边栏搜索框未接线。
   - 证据：`SessionSidebar.tsx` 输入框仅展示 placeholder，未绑定 value/onChange。
2. Search 分页仍是整块替换模型，翻页容易闪烁。
   - 证据：`useSearch.ts` 使用 effect + loading + 请求完成后整体 setResults。
   - 证据：`SearchResultsPanel.tsx` 在 loading 分支直接整块返回“搜索中”。
3. KB 列表存在用户可见“假能力”。
   - 证据：`KnowledgeBaseList.tsx` 批量导出点击后 toast 显示“批量导出功能暂未完全实现”。
4. 浏览器 E2E 覆盖尚未覆盖“搜索分页稳定性”与“会话搜索可用性”两条关键回归。

## 11.2 本次 PR 的设计原则

1. 稳定可用优先于功能扩张。
2. 美观与性能并行，不做“只提速不顾体验”或“只美化不改数据流”。
3. 前端独立闭环，不新增后端接口契约。
4. 所有改动都需有可执行验证（type-check + unit/integration + browser e2e）。

# 12. 本次 PR 执行顺序与交付清单

本节是本次 PR 的执行基线，后续实现与验收严格按此清单推进。

## 12.1 执行顺序（Wave）

### Wave 1：请求层与状态层稳定化

1. Search 数据流迁移到 query cache 模型（基于 React Query）。
   - 查询 key：`query + page + filters`
   - 拆分状态：`isInitialLoading` 与 `isPageFetching`
   - 保留旧数据：`placeholderData`（等价 keepPreviousData 语义）
   - 预取下一页：当前页完成后 prefetch next page
   - 请求取消与乱序保护：切 query/page/filter 时中止旧请求
2. Search API 增加请求取消参数透传。
   - `searchApi.unified` 支持可选 `signal`，由上层 queryFn 注入
3. Chat 会话搜索接线。
   - 在 workspace 层新增 `sessionSearchQuery`
   - 传入 sidebar 受控输入
   - 基于 title 做前端过滤并保留当前会话交互稳定性

### Wave 2：组件层防闪烁与假能力止损

1. `SearchResultsPanel` 改造。
   - 仅首次搜索显示全区 loading
   - 翻页期间保留当前结果，增加局部 loading 提示
2. `SearchPagination` 改造。
   - 翻页请求时按钮局部 loading + `aria-busy`
3. `KnowledgeBaseList` 假能力止损。
   - 移除“暂未完全实现”伪交互
   - 批量导出入口改为禁用并标注“即将上线”或直接隐藏
   - 批量删除能力保持可用

### Wave 3：测试补齐与质量门禁

1. Unit/Integration：补齐会话搜索与搜索分页稳定性测试。
2. Browser E2E：新增两条关键回归。
   - `search-pagination-stability.spec.ts`
   - `chat-session-search.spec.ts`
3. 跑通验证并生成 PR 说明。
   - `npm run type-check`
   - `npm run test:run`

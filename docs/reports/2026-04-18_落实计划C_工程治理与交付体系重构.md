# 1. 计划目标

本计划用于根治“规划很好但落地漂移、分支改了又消失、契约长期并存、验证不足”这一工程组织问题。策略是把治理从文档约束升级为流水线硬约束。

# 2. 根问题定义

1. 规划阶段与实现阶段缺少自动追踪，导致 phase 交付断裂。
2. 分支出现部分 cherry-pick、未完整合并，形成孤岛改动。
3. 契约升级缺少强门禁，fallback 长期遗留。
4. 前后端协同缺少 E2E 阻断条件，后端完成前端滞后。
5. 文档、代码、测试三者缺少强一致校验。

# 3. 治理目标（90 天）

1. Phase 交付可追踪率达到 100%。
2. 未合并分支孤岛改动数量下降到 0。
3. 契约升级均具备强制退役日期与删除里程碑。
4. 关键功能必须有双端 E2E 验证后才允许合并。

# 4. 治理体系重构设计

## 4.1 交付单元化

将每个 phase 拆成可执行交付单元：
1. 目标定义
2. 代码范围
3. 测试范围
4. 文档更新范围
5. 验收清单

每个交付单元必须绑定唯一追踪编号。

## 4.2 分支生命周期制度

分支必须具备以下状态机：
1. created
2. active
3. review-ready
4. merged
5. superseded
6. archived

禁止长期停留在 active 且无更新超过阈值。

## 4.3 契约治理制度

1. 契约变更必须提交迁移文档。
2. fallback 必须声明退役日期。
3. 到期未删除 fallback 时，CI 直接失败。

## 4.4 验证治理制度

1. 单元测试不是唯一门禁。
2. 必须增加跨层 E2E：前端状态 + 后端事件 + 数据层结果一致性。
3. 建立截图与行为回归双轨验证。

# 5. 分阶段执行（10 周）

## 第 1-2 周：追踪框架上线

动作：
1. 建立 phase 追踪台账模板。
2. 每个 PR 关联 phase 交付单元。
3. 增加 PR 模板必填项：交付覆盖、未覆盖、风险。

产出：
1. 追踪台账文档。
2. PR 关联率统计脚本。

## 第 3-4 周：分支治理上线

动作：
1. 盘点历史 feature 分支并分类：可合并、需重基、作废。
2. 对每条分支建立决策记录。
3. 未决策分支禁止继续承载新需求。

产出：
1. 分支治理清单。
2. 分支处置时间表。

## 第 5-6 周：契约门禁上线

动作：
1. 增加契约一致性检查脚本：检测旧字段引用。
2. 增加 fallback 到期检查脚本。
3. CI 中加入 contract-gate 作必过项。

产出：
1. contract-gate 工作流。
2. 违规示例与修复说明。

## 第 7-8 周：E2E 阻断上线

动作：
1. 对 Chat、KB、检索三条关键链路建立端到端用例。
2. 前后端版本矩阵验证，防止单端完成即合并。
3. 将 E2E 结果作为主干合并门禁。

产出：
1. 关键链路 E2E 套件。
2. 失败分级与处置手册。

## 第 9-10 周：审计闭环上线

动作：
1. 建立双周规划 vs 实现审计节奏。
2. 审计结果直连下一迭代待办。
3. 引入治理 KPI 看板。

产出：
1. 审计报告模板。
2. KPI 看板定义与阈值。

# 6. 指标体系

## 6.1 交付一致性指标

1. phase 覆盖率。
2. 规划项落地率。
3. 回滚率。

## 6.2 契约健康指标

1. 旧字段存量。
2. fallback 存活天数。
3. 契约断言触发频次。

## 6.3 发布质量指标

1. 合并后 48 小时故障率。
2. E2E 通过率。
3. 文档同步及时率。

# 7. 风险与回滚

## 7.1 风险

1. 治理项过多导致团队短期效率下降。
2. 门禁太严导致排队拥塞。
3. 历史分支梳理成本高。

## 7.2 缓解

1. 分批启用门禁，先高风险域后全域。
2. 给关键修复开快速通道，但必须补审计记录。
3. 治理脚本先告警后阻断，完成过渡后再强制。

# 8. 资源评估

1. 工程经理 1 人全程负责。
2. 平台工程 1 人主导门禁与脚本。
3. 前后端各 1 人兼职参与治理改造。
4. QA 1 人负责 E2E 体系化。

# 9. 与计划 A/B 的协同

1. 计划 A 的 ChatLegacy 拆分和 SSE 收口，必须纳入契约门禁与 E2E 门禁。
2. 计划 B 的 retrieval contract 强切和 confidence 改造，必须纳入 fallback 退役检查。
3. 三计划统一进入同一治理看板，避免再次出现改动孤岛。

# 10. 退出条件

1. 分支孤岛问题清零。
2. 契约到期 fallback 清零。
3. 双周审计稳定运行 2 个周期。
4. 规划与实现偏差可量化并持续下降。

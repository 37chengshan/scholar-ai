# 1. 计划目标

本计划用于一次性解决前端域内的结构性问题，不采用最小改造路线，而采用“旧壳冻结 + 新内核替换 + 分段切流”的彻底重构路线。

必须一次闭环的核心问题：
1. ChatLegacy 体量过大、业务逻辑混杂、迁移长期停滞。
2. SSE 新旧协议并存，message_id 噪音长期存在，状态机边界不清。
3. Knowledge Base Detail 页面数据、轮询、展示耦合，扩展成本高。
4. UI 组件生态混杂，页面状态反馈体验不统一。
5. 前端测试存在噪音，回归信号纯度不足。

# 2. 范围与边界

## 2.1 在范围内

1. Chat 主流程全量重构（会话列表、消息流、工具轨迹、引用、推理面板）。
2. SSE 协议接入层统一（单入口、单状态机、单事件模型）。
3. Knowledge Base Detail 页面的状态下沉与页面轻量化。
4. 全局交互态统一（loading/empty/error/retry/toast）。
5. 端到端回归链路建设（截图基线 + Playwright 关键路径）。

## 2.2 不在范围内

1. 后端向量检索算法升级（由计划 B 负责）。
2. CI 治理体系重构（由计划 C 负责）。

# 3. 目标架构（终态）

## 3.1 Chat 分层

1. Shell 层：apps/web/src/app/pages/Chat.tsx
2. Workspace 层：apps/web/src/features/chat/workspace/ChatWorkspaceV2.tsx
3. Domain 状态层：apps/web/src/features/chat/state
4. 协议适配层：apps/web/src/features/chat/adapters
5. 展示层：apps/web/src/features/chat/components

原则：
1. 页面层只负责路由参数与容器编排，不持有业务状态。
2. 状态变迁只通过事件驱动，不允许组件直接拼接状态。
3. 所有旧协议只允许存在于 adapter 层，不可渗透到组件。

## 3.2 SSE 单通道模型

事件收敛为统一域模型：
1. STREAM_STARTED
2. ROUTING_DECIDED
3. REASONING_DELTA
4. ANSWER_DELTA
5. TOOL_STATE_CHANGED
6. STREAM_COMPLETED
7. STREAM_FAILED
8. STREAM_CANCELLED

所有 legacy 事件（thought/thinking_status/step_progress）仅在 adapter 内映射，不进入业务层 switch。

# 4. 分阶段执行（6 周）

## 第 0 周：冻结与防扩散

目标：先停损，再重构。

动作：
1. 对 ChatLegacy 与 KnowledgeBaseLegacy 增加禁止新增业务逻辑声明与 lint 规则。
2. 新增变更门禁：触碰 legacy 文件必须附带等价迁移任务链接。
3. 新建前端重构分支并建立每日可合并切片节奏。

产出：
1. Legacy 冻结规则文档。
2. 前端重构任务分解清单（按组件与状态域）。

## 第 1-2 周：Chat 内核拆分

目标：把 1365 行聚合体拆成可测试的 6 个子域。

子域拆分：
1. message-feed
2. citation-panel
3. reasoning-panel
4. tool-timeline
5. composer-input
6. session-sidebar

每个子域必须满足：
1. 单独测试入口。
2. 独立 props 契约。
3. 无跨域隐式读写。

产出：
1. ChatWorkspaceV2 可运行（灰度开关控制）。
2. ChatLegacy 仅保留桥接层。

## 第 3 周：SSE 协议统一

目标：彻底消除双轨并存。

动作：
1. 新建统一 parser：将后端 envelope 规范化为域事件。
2. 统一 message_id 绑定与校验策略：缺失时进入隔离队列并计数，不直接污染主流程。
3. 引入 terminal flush 策略：done/error/cancel 必触发状态收束与缓存落盘。
4. 移除生产代码中的 useSSE 残留，仅保留兼容桥接与迁移测试。

产出：
1. useChatStreamV2。
2. SSE adapter 层覆盖测试。

## 第 4 周：Knowledge Base Detail 彻底解耦

目标：页面去业务化。

动作：
1. 将轮询、导入状态、数据映射下沉到 apps/web/src/features/kb/state。
2. 页面保留查询参数编排与布局拼装。
3. 接口错误、重试、空态全部统一到共享状态组件。

产出：
1. KnowledgeBaseDetailV2。
2. 旧页面保留 1 个版本窗口后下线。

## 第 5 周：视觉与交互统一收口

目标：用户可感知质量提升。

动作：
1. 统一对话区、引用区、推理区层级和排版密度。
2. 统一 skeleton、empty、error、retry、toast。
3. 建立 UI token 收敛表，移除页面内硬编码间距与颜色。

产出：
1. 视觉一致性规范清单。
2. UI 组件替换清单与完成率仪表。

## 第 6 周：切流、下线与稳定期

目标：完成从旧到新的正式切换。

动作：
1. 灰度比例从 10% -> 50% -> 100%。
2. 稳定 3 天后删除 ChatLegacy 主体。
3. 删除旧协议分支与无用 hook。

产出：
1. Legacy 下线变更记录。
2. 切流复盘报告。

# 5. 迁移策略（无感切换）

1. 双容器并行：ChatLegacy 与 ChatWorkspaceV2 并行存在，但新增功能只进 V2。
2. Feature Flag：按用户组和组织维度切流。
3. 数据兼容期：adapter 保留 1 个版本窗口，过期后强制移除。
4. 回归资产：每次切流都执行截图差异与 E2E 核验。

# 6. 风险与回滚

## 6.1 风险

1. 重构期间流式消息偶发重复或丢失。
2. 大规模组件替换导致样式回退。
3. 引用与推理区域交互异常。

## 6.2 回滚策略

1. 保留旧容器开关，支持 5 分钟内全量回切。
2. 协议层异常时回退到稳定 parser 版本。
3. 每个阶段仅允许单一主题上线，避免多风险叠加。

# 7. 验收矩阵

## 7.1 功能验收

1. Chat 全流程：发送、流式、中断、重试、确认、恢复，全通过。
2. KB Detail：导入、轮询、筛选、详情跳转，全通过。
3. SSE：message_id mismatch 告警下降到每 10 万事件不超过 1 次且可追踪。

## 7.2 质量验收

1. 类型检查零错误。
2. 单元测试通过率 100%。
3. 关键 E2E 场景通过率 100%。
4. 截图基线差异仅允许白名单变化。

## 7.3 体验验收

1. 首屏交互就绪时间下降至少 20%。
2. 聊天关键操作点击到可见反馈小于 300ms。
3. 错误可恢复率达到 95% 以上。

以下条件全部满足方可宣布计划 A 完成：
1. ChatLegacy 主体下线。
2. SSE 双轨事件清零（仅 adapter 中存在历史映射且已标记退役日期）。
3. KB Detail 页面仅保留编排逻辑。
4. 核心页面截图基线稳定 2 个发布周期。

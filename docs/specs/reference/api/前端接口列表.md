## Login（登录页）

### API 接口列表

| 方法 | 路径 | 功能 | 请求体/参数 |
|------|------|------|------------|
| POST | `/api/v1/auth/login` | 用户登录（Cookie-based） | `username={email}&password={password}` (form-urlencoded) |
| POST | `/api/v1/auth/register` | 用户注册 | `{ email, password, name }` (JSON) |

**说明**：
- 登录通过 `useAuth().login()` 间接调用 `authApi.login()`，注册直接调用 `authApi.register()`
- 两者均使用 `@/utils/apiClient`（axios 实例），启用 `withCredentials` 发送 HttpOnly Cookie
- 注册成功后自动调用登录并跳转到仪表盘

### 导航逻辑问题

- **[死按钮] SSO 登录按钮**（第 351-353 行）：`onClick` 未绑定任何处理函数，点击无响应。按钮文本为 `{t.sso}`（"单点登录 (SSO)" / "SSO Login"），属于占位UI。
- **[死按钮] 申请访问权限按钮**（第 354-356 行）：`onClick` 未绑定任何处理函数，点击无响应。按钮文本为 `{t.requestAccess}`（"申请访问权限" / "Request Access"），属于占位UI。
- **[可优化] 注册成功后双重跳转**：注册流程为 `register()` → `login()` → `navigate("/dashboard")`。若 `login()` 抛出异常，错误被捕获后仅显示错误提示，但此时用户可能已在后端创建成功，造成困惑。建议注册成功后只提示"注册成功，请登录"而非自动登录跳转。
- **[可优化] 忘记密码链接**（第 314 行）：导航到 `/forgot-password` 路由，需确认该页面是否已完成实现。
- **[潜在竞态] Auth 状态检测时机**：第 44-48 行 `useEffect` 监听 `isAuthenticated` 和 `authLoading`，当 `authLoading` 从 `true` → `false` 且 `isAuthenticated` 为 `true` 时跳转仪表盘。此逻辑依赖 AuthContext 内部状态时序，若 `login()` 调用完成但 Context 未及时更新，可能导致导航延迟或异常。

## Register（注册页）

### API 接口列表

| 方法 | 路径 | 功能 | 请求体/参数 |
|------|------|------|------------|
| POST | /api/v1/auth/register | 注册新用户 | `{email: string, password: string, name: string}` |
| POST | /api/v1/auth/login | 注册后自动登录 | `FormData: username=email, password` |

**源码位置**: `/scholar-ai/apps/web/src/services/authApi.ts`

**关键代码** (Register.tsx 第 57-88 行):
```typescript
const handleRegister = async (e: React.FormEvent) => {
  e.preventDefault();
  setError("");

  // Validate passwords match
  if (password !== confirmPassword) {
    setError(isZh ? "两次输入的密码不一致" : "Passwords do not match");
    return;
  }

  // Validate password requirements
  const passwordRegex = /^(?=.*[A-Z])(?=.*[a-z])(?=.*\d).{8,}$/;
  if (!passwordRegex.test(password)) {
    setError(isZh ? "密码不符合要求" : "Password does not meet requirements");
    return;
  }

  setIsLoading(true);

  try {
    // Call register API
    await authApi.register(email, password, name);

    // Auto-login after successful registration
    await login(email, password);
    navigate("/dashboard");
  } catch (err: any) {
    const errorMessage = err.message || (isZh ? "注册失败" : "Registration failed");
    setError(errorMessage);
  } finally {
    setIsLoading(false);
  }
};
```

### 导航逻辑问题

- **注册成功后直接跳转 dashboard，无注册成功提示**: 用户视角：注册成功后没有任何"注册成功"或"欢迎"等提示信息，直接跳转到 `/dashboard`。从用户旅程角度，缺少一个"注册成功"的过渡页面或 toast 提示，用户可能不清楚自己的注册是否真的成功了。

- **缺少登录入口的直接导航**: 页面底部有"已有账户？登录"链接（第 291-298 行），但点击后跳转到 `/login`，如果登录页不存在会导致 404。

- **未登录状态下直接访问注册页**: 如果用户已登录（Cookie 有效），直接访问 `/register` 仍然显示注册表单，未做重定向检查。建议在组件 mount 时检查认证状态，已登录用户应重定向到 dashboard。

- **注册失败后表单数据未保留问题**: 当注册 API 调用失败时，`email`/`name`/`password`/`confirmPassword` 状态不会清空，这是正确行为；但如果用户修改表单后提交失败，原有输入会保留。

- **缺少邮箱格式的即时校验**: Email 输入框使用 `type="email"` 和 `required`，但浏览器原生校验和业务逻辑校验都只在提交时触发。未实现 `onBlur` 或 `onChange` 即时校验，用户无法立即得知邮箱格式错误。

## ImportJob（批量导入）

### API 接口列表

| 方法 | 路径 | 功能 | 请求体/参数 |
|------|------|------|------------|
| POST | `/api/v1/knowledge-bases/:kbId/imports` | 创建单个导入任务 | `{ sourceType, payload, options? }` |
| PUT | `/api/v1/import-jobs/:jobId/file` | 上传文件到导入任务 | `FormData: file` |
| GET | `/api/v1/import-jobs/:jobId` | 获取单个导入任务详情 | Path: jobId |
| GET | `/api/v1/import-jobs?knowledgeBaseId=:kbId` | 获取 KB 的导入任务列表 | Query: knowledgeBaseId, status?, limit?, offset? |
| POST | `/api/v1/import-sources/resolve` | 解析单个来源（预览） | `{ sourceType, input }` |
| POST | `/api/v1/import-sources/resolve-batch` | 批量解析来源 | `{ items: string[] }` |
| POST | `/api/v1/knowledge-bases/:kbId/imports/batch` | 批量创建导入任务 | `{ items: CreateImportRequest[] }` |
| POST | `/api/v1/import-jobs/:jobId/retry` | 重试失败任务 | `{ retryFromStage?: string }` |
| POST | `/api/v1/import-jobs/:jobId/cancel` | 取消任务 | 无 |
| POST | `/api/v1/import-jobs/:jobId/dedupe-decision` | 提交去重决策 | `{ decision: string, matchedPaperId?: string }` |
| GET | `/api/v1/import-jobs/:jobId/stream` | SSE 任务进度流 | Path: jobId（返回 EventSource URL） |
| GET | `/api/v1/knowledge-bases/:kbId/upload-history` | 获取 KB 上传历史 | Query: limit?, offset? |

**说明**：
- ImportJob API 通过 `importApi.ts` 封装
- sourceType 可选值：local_file / arxiv / pdf_url / doi / semantic_scholar
- Phase B external import 主链：
  - Search 通过 `/api/v1/search/unified` 获取 external results
  - 单篇导入统一走 `/api/v1/knowledge-bases/:kbId/imports`
  - 批量导入统一走 `/api/v1/knowledge-bases/:kbId/imports/batch`
- `resolve()` 用于导入前预览（解析来源获取摘要信息）
- `resolveBatch()` 批量预览多个来源
- `submitDedupeDecision()` 的 decision 可选值：reuse_existing / import_as_new_version / force_new_paper / cancel
- SSE stream 通过 `getStreamUrl(jobId)` 获取 EventSource URL
- 状态分层：
  - `availability`: `metadata_only | pdf_available | pdf_unavailable`
  - `libraryStatus`: `not_imported | importing | imported_metadata_only | imported_fulltext_ready`
  - `job.stage`: `queued | resolving | downloading | parsing | indexing | completed_metadata_only | completed_fulltext_ready | failed | cancelled`

**源码路径**：`/scholar-ai/apps/web/src/services/importApi.ts`

# File Metadata Contract

> Version: v1.0 ┬Ę Backend Pipeline Cleanup Phase

## 1. Authoritative Source

`ImportJob.storage_key` is the **single authoritative source** of truth for a
file's location.  `Paper.storage_key` and `ProcessingTask.storage_key` are
**copies** synchronised from `ImportJob` by the pipeline; they must never be
generated independently.

## 2. Canonical `storage_key` Format

```
uploads/{user_id}/{YYYY}/{MM}/{DD}/{import_job_id}.pdf
```

| Segment | Example | Notes |
|---------|---------|-------|
| `uploads/` | literal prefix | distinguishes upload objects from other storage namespaces |
| `{user_id}` | `usr_abc123` | UUID or opaque user identifier |
| `{YYYY}/{MM}/{DD}` | `2026/04/26` | UTC date of the upload |
| `{import_job_id}` | `imp_xyz789` | ImportJob primary key |
| `.pdf` | literal suffix | only PDF uploads are supported |

**Generated by:** `import_file_service.build_upload_storage_key(user_id, import_job_id)`.

**Prohibited patterns:**
- `{user_id}/{YYYYMMDD}/{uuid}.pdf` (old papers/upload format ŌĆō removed)
- Any format that does not embed the `import_job_id`

## 3. FileMetadata Fields

The following fields are considered part of the file metadata contract and are
stored on `ImportJob` as the authoritative record:

| Field | Column | Type | Source |
|-------|--------|------|--------|
| User identifier | `import_jobs.user_id` | str | auth token |
| Storage key | `import_jobs.storage_key` | str | `build_upload_storage_key()` |
| SHA-256 hash | `import_jobs.file_sha256` | str | `hashlib.sha256(content).hexdigest()` |
| File size | `import_jobs.size_bytes` | int | `len(content)` |
| Original filename | `import_jobs.filename` | str | `UploadFile.filename` |
| MIME type | `import_jobs.mime_type` | str | `"application/pdf"` (hardcoded) |
| Created timestamp | `import_jobs.created_at` | datetime | server UTC now |

## 4. Propagation Rules

1. `ImportJob` is created first; `Paper` and `ProcessingTask` do not exist yet.
2. After the pipeline materialises the `Paper`, `ImportJob.paper_id` is set.
3. `Paper.storage_key` ŌåÉ copied from `ImportJob.storage_key` by `import_worker`.
4. `ProcessingTask.storage_key` ŌåÉ copied from `ImportJob.storage_key` when the
   task is created inside `import_worker`.
5. No other code path is allowed to set `Paper.storage_key` or
   `ProcessingTask.storage_key` to a different value.

## 5. Storage Root Configuration

All local file reads and writes must use `settings.LOCAL_STORAGE_PATH` (defined
in `apps/api/app/config.py`).  Direct use of `os.getenv("LOCAL_STORAGE_PATH")`
is prohibited; this was enforced in Pipeline Cleanup v1.

Helper: `import_file_service.local_storage_root()` returns
`Path(settings.LOCAL_STORAGE_PATH).resolve()`.

## 6. Path Traversal Protection

`ObjectStorage._get_local_path()` validates that the resolved path is still
under the storage root before any read or write.  This check must be preserved
when modifying storage helpers.

## 7. Upload Session Protocol

Multi-part uploads via the `UploadSession` API follow the same contract:
- Part chunks are written to `{LOCAL_STORAGE_PATH}/sessions/{session_id}/parts/`
- On complete, assembled file is moved to the canonical `storage_key` path
- `UploadSessionService._upload_storage_key()` delegates to
  `build_upload_storage_key()` to ensure format consistency

## 8. Future: FileRecord Model

When a dedicated `FileRecord` database model is introduced it must:
- Use `storage_key` as a unique index (natural key)
- Reference `import_job_id` as a foreign key
- Not replace `ImportJob.storage_key` but extend it with additional metadata

The introduction of `FileRecord` does not require any changes to the storage
key format defined in section 2.
